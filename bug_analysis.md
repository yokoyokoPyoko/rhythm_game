## 調査結果

### 1. 原因箇所①（最重要）
**`src/index.ts` : 631〜636行**

```ts
const beatIndex = Math.floor(songTime / beatMs);
if (beatIndex >= nextClickBeat && beatIndex > 0) {
  metronome();
  nextClickBeat = beatIndex + 1;
}
```

**なぜ起きるのか**
メトロノーム音は `requestAnimationFrame`（描画ループ）の中で「今のビート番号が変わったら鳴らす」という後追い検出で発火している。`requestAnimationFrame` は描画タイミングに依存するため、実際のビートより数十ms〜100ms以上遅れて発音される。`audioCtx.currentTime` でスケジューリングせず即時の `o.start(t)` を使っているため、音符の判定タイミング（`targetBeat * beatMs`）と聴こえるメトロノーム音の発音時刻がズレており、「SPACEを押すべきタイミング」と「ビート音が聴こえるタイミング」が一致しない。

**修正の方向性**
メトロノームを `requestAnimationFrame` ループで後追い検出するのではなく、`audioCtx.currentTime` を基準に先行スケジューリング（lookahead scheduler）に切り替える。

### 2. 原因箇所②
**`src/index.ts` : 305行（音波サバイバル）**

```ts
beatPulse = 1 - (songTime % beatMs) / beatMs;
```

**なぜ起きるのか**
`beatPulse` はビート「直後」に1になり「直前」に0になる逆位相の値だが、視覚フラッシュ（338〜340行）は「ビート直後に大きく光り → 次ビート直前に消える」という正しい周期に見える。ただし **メトロノーム音のズレ（原因①）** によって視覚と聴覚がそもそも同期していないため、視覚的フィードバックも「ずれた音」に合わせて光ることになり、脳がリズムとして認識できない。

**修正の方向性**
原因①のlookaheadスケジューラと同じタイムラインに視覚パルスも同期させる（`audioCtx.currentTime` ベースの位相で算出する）。

### 3. 原因箇所③
**`src/index.ts` : 74行 + 60〜72行**

```ts
function metronome() { click(880, 0.05, 0.15, "square"); }
// click内: o.start(t) で t = audioCtx.currentTime（即時）
```

**なぜ起きるのか**
880Hz 方形波を 50ms という極めて短い時間で鳴らしており、`exponentialRampToValueAtTime` で急減衰するため、聞こえるのはクリック音というよりほぼ「ポッ」という微小な音になる。音量も `vol=0.15` と控えめで、BGMや効果音が重なると埋もれ、プレイヤーがビートを「聴き取れない」状態になっている。

**修正の方向性**
メトロノーム音の持続時間を 80〜120ms 程度に伸ばし、音量を 0.3〜0.4 に上げ、アタックを明確に感じられる波形（`square` のまま）で鳴らす。

### まとめ

| 優先度 | 箇所 | 根本原因 |
|--------|------|----------|
| 🔴 最重要 | L631〜636 | rAFループによる後追い発音 → 音符タイミングと音がズレる |
| 🟡 重要 | L305, L575 | 視覚パルスが「ずれた音」に同期してしまう |
| 🟠 補助 | L74, L60〜72 | メトロノーム音が短すぎ・小さすぎて聞こえない |

**本質的な問題は原因①です**。リズムゲームの根幹である「音符を叩くべき瞬間」と「ビート音が聞こえる瞬間」が `requestAnimationFrame` の遅延によって常にズレており、プレイヤーは正しいタイミングを音から学習できない状態になっています。
